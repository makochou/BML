# 新建 API 账号弹窗重设计说明

## 1. 变更目标

本次改造的目标不是简单“把空白页补满”，而是将 `新建账号` 弹窗恢复为一套正式可交付的开通工作台，并同时满足以下要求：

1. 视觉层要明显优于普通表单弹窗，能够体现“开通交付台”的仪式感和层级感。
2. 业务层不能分叉，继续复用统一字段 Schema、统一表单校验、统一提交入口。
3. 结构层要足够清晰，便于后续新增字段、扩展卡片或抽组件复用。
4. 交互层必须保留原有拖拽、缩放、全屏、滚动等弹窗治理能力。

## 2. 涉及文件

- 页面主文件：`bml-frontend/src/views/api/ApiAccountManage.vue`
- 白名单编辑组件：`bml-frontend/src/components/api-account/ApiEnvironmentWhitelistEditor.vue`
- 表单字段 Schema：`bml-frontend/src/composables/useApiAccountFormSchema.ts`
- 表单校验与负载构建：`bml-frontend/src/composables/useApiAccountFormValidation.ts`

## 3. 最终结构

### 3.1 整体分层

当前 `新建账号` 弹窗由四层组成：

1. 标题层：继续使用弹窗标题栏，保留拖拽、全屏、关闭按钮与层级标记。
2. 背景装饰层：使用 `EmptyWorkbenchCanvas` 作为纯视觉底布，负责氛围，不承载业务内容。
3. 开通工作台层：真实的新建内容，包括 Hero、侧栏卡片、表单区、白名单区和底部交付区。
4. 提交结果层：创建成功后仍旧进入统一的凭证交付弹窗，不重新发明提交流程。

### 3.2 新建态布局

当前新建页采用“顶部 Hero + 下方双栏工作台”的结构：

1. 顶部 Hero：展示开通说明、交付卖点和实时摘要卡。
2. 左侧侧栏：展示账号画像、三步开通节奏、三环境白名单概览。
3. 右侧主区：展示统一 Schema 渲染的业务字段和环境白名单编辑器。
4. 底部交付区：展示保存说明、风险提示和操作按钮。

这样设计的原因：

1. 顶部 Hero 负责建立氛围和重点信息，不再让页面显得“像空白表单”。
2. 左侧侧栏负责承载辅助信息，避免把说明文案继续塞进表单字段之间。
3. 右侧主区保持高密度录入效率，首屏即可看到足够多的核心字段。
4. 统一底部交付区能让“取消 / 重置 / 创建”动作收口，交互节奏更稳定。

## 4. 关键实现方式

### 4.1 新建态与编辑态彻底分支

当前模板继续使用：

1. `isCreateMode === true`：走 `teleport + 自定义固定面板` 的新建工作台布局，不再复用 Arco 默认 `a-modal` 的 body / wrapper 滚动模型。
2. `isCreateMode === false`：继续沿用原有编辑治理台布局与 `a-modal`。

这样做的价值：

1. 新建设计可以大胆重做，不会影响编辑页既有体验。
2. 编辑态的侧栏、概览卡、页脚逻辑完全不需要回归测试式重构。
3. 两者继续共享同一份表单模型与提交流程，避免接口契约分叉。
4. 新建态的标题固定、内容内滚和右侧内部滚动条可以完全由页面自身控制，不再受第三方弹窗默认滚动实现干扰。
5. 创建弹层继续保留“拖动位置 + 四边四角拉伸缩放”的桌面窗体式交互体验。

### 4.2 视觉背景与业务内容分离

当前 `account-create-viewport` 中同时存在两类内容：

1. `EmptyWorkbenchCanvas`：纯背景装饰层，绝对定位，`pointer-events: none`。
2. `account-create-content-shell`：第①层内部真实内容滚动壳，统一包裹第②层、第③层全部内容，并直接承载右侧原生纵向滚动条。

这样处理的好处：

1. 背景可以随时换主题，不需要碰业务表单结构。
2. 滚动、输入、点击全部由业务层承载，不会受到装饰层影响。
3. 后续如果需要把视觉底布抽成复用组件，改动范围会非常小。
4. 滚动职责集中在一个内容壳上，后续新增区块也不容易再把滚动分散到外层容器。

### 4.3 统一配置模型

为了避免在模板中堆大量分支判断，本次将新建态辅助信息收口为几组配置化数据：

1. `accountCreateHeroTags`
2. `accountCreateOverviewStats`
3. `accountCreateIdentityFacts`
4. `accountCreateFlowSteps`
5. `accountCreateEnvironmentCards`
6. `accountCreateFooterTips`

维护原则：

1. 改文案优先改这些常量/计算属性。
2. 改口径优先改计算属性，不直接在模板中写条件拼接。
3. 模板只负责渲染结构，不负责承担业务判断。

### 4.4 统一字段与统一提交

当前新建页没有新增任何接口层分叉逻辑，仍然使用：

1. `useApiAccountFormSchema()` 统一定义字段结构、组件类型与辅助文案。
2. `useApiAccountFormValidation()` 统一处理字段校验、回调地址校验、系统编码标准化、白名单去重等逻辑。
3. `submitAccountForm()` 统一处理创建与编辑提交。

这保证了：

1. 新建设计升级不会引入新的数据口径偏差。
2. 新建和编辑不会出现“前端看起来一致，但提交结构不一致”的问题。
3. 后续新增普通字段时仍然优先改 Schema，而不是去模板里手写 `a-form-item`。

## 5. 滚动与弹窗约束

当前新建态滚动规则如下：

1. `account-create-viewport` 继续固定新建页整体视觉区域。
2. `account-modal--create` 作为第①层弹窗卡片本身，只负责固定边界与承载标题层。
3. `account-modal-body--create` 作为第①层固定内容视口，不再直接承担浏览器原生滚动。
4. `account-create-content-shell` 作为第①层内部唯一真实纵向滚动区，统一显示第②层、第③层全部内容。
5. 右侧纵向滚动条直接由 `account-create-content-shell` 原生显示，不再通过额外脚本同步独立滚动条。
6. 顶部 Hero、侧栏卡片、表单区、白名单区、底部交付区全部直接塞回第①层内部内容区显示。

这样做可以避免：

1. 内容继续把弹窗本体向下撑出浏览器视口。
2. 弹窗外层、内层、局部区域同时出现多套纵向滚动条。
3. 第②层、第③层与第①层出现滚动职责分叉，导致交互焦点混乱。
4. 新增字段后底部按钮跑出可视区之外。

### 5.1 一级标题层固定规则

本次进一步补充了“第②层、第③层内容全部并入第①层显示，且右侧滚动条明确出现在第①层内部内容区”的约束，具体实现如下：

1. 弹窗头部 `.arco-modal-header` 显式设置为 `sticky + top: 0`。
2. 弹窗打开时，通过 `syncAccountModalPageScrollLock()` 将背景页面冻结在当前滚动位置。
3. 冻结时把页面原始滚动位置保存到 `accountModalPageLockState.scrollTop`。
4. 同时通过 `syncAccountModalContainerScrollLock()` 直接锁定业务页根容器、`page-container` 与 Layout 内容区等真实滚动节点。
5. 编辑态仍通过 `syncAccountModalWrapperScrollLock()` 锁定 Arco 自带的 `.arco-modal-wrapper`；新建态则直接绕过该层，改由自定义固定面板承载。
6. 新建态移除原先独立的 `account-create-scroll-region`，改为仅保留 `account-create-content-shell` 这一套内部原生滚动结构。
7. `account-create-content-shell` 直接使用原生 `overflow-y: scroll`，强制在第①层内部显示右侧滚动条。
8. `account-create-viewport` 与 `account-create-content-shell` 显式拦截滚轮事件冒泡，避免滚动在边界时继续传给外层页面。
9. 弹窗关闭时移除页面锁，并将背景页滚动位置精确恢复。

这样处理后可以保证：

1. 第②层与第③层内容全部在第①层内部显示，不再有独立滚动职责。
2. 右侧纵向滚动条明确出现在第①层内部内容区右侧，由内容壳自身承载。
3. 无论滚动的是鼠标滚轮还是触控板，交互焦点都停留在第①层内部。
4. 标题层固定，内容层滚动，层级关系清晰。
5. 即使项目布局采用“内容区内部滚动”而不是 `body` 滚动，标题层仍然稳定。
6. 即使用户滚动的是 Arco Modal 外层 wrapper，整块弹窗也不会再被向上推走。
7. 关闭弹窗后，用户仍能回到打开弹窗前的原页面位置，不会丢失上下文。

### 5.2 拖拽与缩放规则

当前创建弹层继续复用统一的 `accountModalViewport` 视口状态，但在自定义弹层上补齐了显式交互入口：

1. 顶部标题栏保留拖动能力，按住标题层空白区域即可移动窗口位置。
2. 标题操作区增加轻量化拖拽 grip 图标，用于提示“该弹层支持拖动 / 缩放”，但不再使用大面积提示条破坏标题区视觉。
3. 四边与四角保留缩放热区，用户可直接向上、下、左、右以及四个对角方向拉伸窗口尺寸；角部只在悬浮或缩放时显示弱化提示，不做常驻大把手。
4. 缩放命中优先识别显式手柄，其次才回退到边缘热区判断，兼顾可发现性与容错。
5. 位置与尺寸仍统一写入 `accountModalViewport`，编辑态与创建态共用同一套边界约束与全屏切换规则。
6. 全屏态会自动隐藏缩放手柄，避免与全屏布局发生冲突。
7. 创建态窗口位置改为直接输出 `left / top / width / height`，不再依赖居中 `transform` 偏移，因此拖动更顺畅，也可以精准贴合浏览器四边。
8. 创建态标题拖动与缩放手柄改为模板内显式事件绑定，不再依赖运行时查询 DOM 绑定事件，能降低事件命中不稳定和边缘拖不动的问题。

维护建议：

1. 如果后续要把这套拖拽缩放骨架复用到其他自定义治理弹层，优先复用 `accountModalViewport`、指针状态与缩放手柄配置，不要重新发明一套拖拽状态机。
2. 如果要调整最小宽高，只改 `ACCOUNT_MODAL_MIN_WIDTH` 与 `ACCOUNT_MODAL_MIN_HEIGHT`，不要在样式里单独写死尺寸阈值。

## 6. 样式设计原则

### 6.1 视觉方向

本次新建设计采用“高对比深色 Hero + 浅色玻璃卡片”的组合：

1. 顶部 Hero 使用蓝青渐变、网格纹理与柔光层，负责开场氛围。
2. 侧栏与表单区使用浅色玻璃卡片，保证可读性和精致感。
3. 按钮、标签、状态块统一沿用蓝青方向的渐变与胶囊语言。

### 6.2 命名约束

所有新建设计样式统一使用 `account-create-*` 前缀，目的是：

1. 避免污染编辑态样式。
2. 避免未来其他治理弹窗误命中这些规则。
3. 让后续维护者能够通过前缀快速定位新建工作台相关样式。

### 6.3 响应式策略

在中小屏下统一降级为：

1. 顶部 Hero 由双栏变单栏。
2. 左侧侧栏与右侧主区由双栏变纵向堆叠。
3. 底部操作区改为纵向排列，避免按钮拥挤换行。
4. 表单壳层标题区由横向布局改为纵向布局，保证说明文案完整展示。

## 7. 后续维护教程

### 7.1 如果要改顶部摘要卡

优先修改：

- `accountCreateOverviewStats`

适合修改的内容：

1. 卡片数量
2. 主数值
3. 提示文案
4. 色调

### 7.2 如果要改左侧开通步骤

优先修改：

- `accountCreateFlowSteps`

建议：

1. 不要在模板里直接写新的步骤 HTML。
2. 保持每一步只表达一个阶段目标，避免描述过长。

### 7.3 如果要改表单字段

优先修改：

- `bml-frontend/src/composables/useApiAccountFormSchema.ts`

标准步骤：

1. 在对应分区的 `fields` 中新增字段。
2. 设置 `kind`、`required`、`componentProps`、`helper`。
3. 如有格式校验，再到 `useApiAccountFormValidation.ts` 增补规则。

### 7.4 如果要改环境白名单展示

优先修改：

- `ApiEnvironmentWhitelistEditor.vue`
- `accountCreateEnvironmentCards`

维护原则：

1. 编辑器负责输入与字段交互。
2. 左侧环境概览卡负责摘要展示。
3. 摘要卡只读，不要在摘要卡里塞输入交互。

## 8. 验证方式

本次修改完成后，已在 `bml-frontend` 目录执行：

```bash
npm run build
```

构建通过，说明以下内容均已通过编译校验：

1. Vue 模板语法
2. TypeScript 类型
3. 组合式计算属性引用
4. 样式作用域与类名映射

## 9. 后续建议

如果后续还要继续演进这套新建工作台，建议按以下顺序推进：

1. 抽出“创建工作台侧栏卡片”业务组件，复用到其他接入账号开通场景。
2. 抽出“交付页脚”通用组件，统一保存说明、操作按钮与风险提示。
3. 继续把 Hero 摘要卡沉淀为配置化卡片模型，减少页面级模板体积。
