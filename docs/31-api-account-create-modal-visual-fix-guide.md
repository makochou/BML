# 新建 API 账号弹窗重设计说明

## 1. 本次目标

本次重构将“新建 API 账号”弹窗从临时空白页恢复为正式开通工作台，并保留以下交互能力：

1. 支持标题栏拖拽移动。
2. 支持拖拽四边和四角调整窗口大小。
3. 支持拖到贴合浏览器边界，也支持标题栏右上角全屏切换。
4. 支持在弹窗内部纵向滚动查看全部新建内容。

## 2. 实现文件

- 页面主文件：`bml-frontend/src/views/api/ApiAccountManage.vue`
- 白名单编辑组件：`bml-frontend/src/components/api-account/ApiEnvironmentWhitelistEditor.vue`
- 表单字段 schema：`bml-frontend/src/composables/useApiAccountFormSchema.ts`
- 表单校验与提交负载构建：`bml-frontend/src/composables/useApiAccountFormValidation.ts`

## 3. 页面结构说明

### 3.1 顶部展示区

当前版本已经去掉顶部大 Hero 展示区，原因是：

1. 顶部视觉占高过大，会压缩真正可填写字段的首屏可见面积。
2. 新建场景更需要快速进入表单，而不是先看大段说明。
3. 关键提示已经收口到底部交付区和左侧治理卡片中，不再需要额外的大横幅承载。

### 3.2 主表单区

当前版本已经进一步去掉左侧整列治理侧栏，原因是：

1. 新建场景最核心的是录入字段，左侧信息卡会持续挤占横向空间。
2. 开通节奏、环境摘要和填写建议都属于辅助信息，不应压过主表单本身。
3. 去掉侧栏后，表单和白名单编辑器可以获得更充足的可视宽度，首屏字段更完整。

当前新建页只保留主表单区，继续复用统一 schema 驱动表单与统一白名单组件：

1. `GovernanceFormSections` 负责渲染基础归属、安全策略、回调说明等字段。
2. `ApiEnvironmentWhitelistEditor` 负责环境白名单编辑。
3. 底部交付区集中展示保存说明、重置、取消、创建操作。

滚动规则：

1. 新建弹窗的背景布本身是唯一纵向滚动层。
2. 表单内容必须始终包裹在弹窗背景布内部显示。
3. 当字段超出可视高度时，通过弹窗内部上下滚动查看，不允许内容溢出到弹窗外部。
4. 当前实现进一步拆成 `account-create-viewport + account-create-scroll-region` 两层，确保弹窗背景布尺寸固定不跟随内容高度变化。

这样做的好处是：

1. 视觉层重做，但表单字段定义和校验逻辑不需要分叉。
2. 新增字段时优先改 schema，不需要重复改模板结构。
3. 后续如果其他治理页面需要复用这套布局，可以直接抽主表单容器和底部交付区。

## 4. 关键代码设计

### 4.1 新建态和编辑态分离

当前弹窗模板明确拆分为两个分支：

1. `isCreateMode === true`：使用新的开通工作台布局。
2. `isCreateMode === false`：继续使用原来的编辑治理工作台布局。

这样可以保证：

1. 新建页可以大胆重设计，不影响编辑页。
2. 新建与编辑共享同一份表单数据结构和提交逻辑。

### 4.2 统一配置模型

新建页新增了几组配置化数据：

1. `accountCreateFlowSteps`
2. `accountCreateFooterTips`
3. `accountCreateIdentityFacts`
4. `accountCreateEnvironmentCards`

这些数据全部通过常量或计算属性生成，避免在模板里堆业务判断，方便后续复用和维护。

### 4.3 统一校验与统一提交

提交仍然走 `submitAccountForm()`，并依赖：

1. `useApiAccountFormSchema()` 统一字段定义。
2. `useApiAccountFormValidation()` 统一字段校验、标准化和 payload 构建。

这保证了：

1. 新建页虽然重做视觉，但不会引入接口层行为偏差。
2. 业务系统编码大写化、回调地址校验、白名单去重等逻辑仍然只有一份。

### 4.4 滚动承载约束

为避免新建页后续字段被内容撑出弹窗底部，当前实现明确约束：

1. `account-modal-body--create` 负责新建态的纵向滚动。
2. `account-create-viewport` 负责锁定新建弹窗背景布的可视区域。
3. `account-create-scroll-region` 作为新建页内部滚动承载层，专门负责内容区上下滚动。
4. 新建态内部容器不再使用会把内容向外撑开的最小高度策略。
5. 底部交付区按正常文档流排列，跟随弹窗内容一起在背景布内部显示。

## 5. 样式设计原则

### 5.1 视觉方向

本次新建设计使用“深色渐变 Hero + 浅色玻璃卡片”的组合：

1. 顶部负责氛围和重点信息。
2. 下方负责高密度字段录入。
3. 通过圆角、渐变、柔和投影、状态胶囊统一风格。

### 5.2 可维护性

样式命名统一使用 `account-create-*` 前缀，避免污染编辑态或其他弹窗。

### 5.3 响应式

在中小屏下：

1. Hero 改为单列。
2. 左右布局改为上下布局。
3. 底部操作区改为纵向排列。

## 6. 后续扩展建议

如果后续还要继续演进，建议按下面顺序做：

1. 抽出“治理侧栏卡片”通用组件，供新建、编辑、审批页面复用。
2. 抽出“弹窗底部交付区”通用样式，统一按钮节奏与交付提示。
3. 将 Hero 指标卡与环境摘要卡沉淀为可配置卡片模型。

```bash
npm run build
```

构建通过，说明模板、样式、类型和脚本逻辑均已通过编译校验。
