# 编码规范与中文注释规范

## 1. 目标

本规范用于解决以下问题：

- 中文注释或文档出现乱码
- 不同编辑器保存后编码不一致
- 团队成员在 Windows / macOS / Linux 之间协作时换行符漂移
- Java、SQL、Markdown、YAML 等文本文件混用不同编码

## 2. 全仓统一规则

### 2.1 文件编码

除非文件格式有特殊要求，仓库内所有文本文件统一使用：

- 编码：`UTF-8`
- 换行：`LF`
- 文件结尾：保留换行

Windows 脚本保留 `CRLF`：

- `*.ps1`
- `*.cmd`
- `*.bat`

### 2.2 规范入口

本仓库已经新增以下治理文件：

- 根目录 `.editorconfig`
- 根目录 `.gitattributes`

作用分别是：

- `.editorconfig`：约束编辑器保存规则
- `.gitattributes`：约束 Git 的文本归一化规则

## 3. 注释编写规范

### 3.1 注释必须服务理解成本

只有以下场景才建议写注释：

- 业务规则不直观
- 安全约束不明显
- 兼容逻辑或历史包袱必须解释
- 非常规实现需要说明原因

### 3.2 不要写低价值注释

禁止这类注释：

- “给变量赋值”
- “调用方法”
- “返回结果”

这类注释只会增加噪声，不会提升可维护性。

### 3.3 推荐注释风格

推荐写法：

```java
// OpenAPI 请求体需要先缓存，后续签名校验才能读取原始 body。
```

```java
/**
 * 统一处理系统级异常，确保所有入口返回同一套协议结构。
 */
```

### 3.4 文档注释要求

对外接口、基础设施类、公共工具类建议补齐 Javadoc，至少说明：

- 作用
- 输入
- 输出
- 边界条件

## 4. 开发环境要求

### 4.1 IntelliJ IDEA

建议检查以下设置：

1. `Settings -> Editor -> File Encodings`
2. 全局编码设为 `UTF-8`
3. 项目编码设为 `UTF-8`
4. `Transparent native-to-ascii conversion` 关闭

### 4.2 VS Code

建议检查以下设置：

1. `files.encoding = utf8`
2. `files.eol = \n`
3. Java、Markdown、YAML 不要单独改成 GBK

## 5. 常见乱码排查

### 5.1 现象一：Git 里正常，IDE 里乱码

通常是编辑器按错误编码打开文件。优先检查 IDE 的文件编码设置。

### 5.2 现象二：本地正常，提交后同事打开乱码

通常是文件被错误地按 ANSI / GBK 保存，或者缺少 `.editorconfig` / `.gitattributes` 约束。

### 5.3 现象三：Windows 脚本执行异常

优先检查：

- 是否误把 `*.ps1` 保存成 `LF`
- 是否误把 `*.cmd` 保存成 `UTF-8 BOM`

## 6. 新增文件检查清单

新增任何文本文件前，请自检：

1. 是否使用 UTF-8 编码
2. 是否符合 `.editorconfig`
3. 是否存在中文乱码
4. 注释是否真正有信息量
5. 是否与现有统一响应、统一异常、统一命名规范一致

## 7. 维护要求

- 任何发现乱码的文件，修复后都必须以 UTF-8 重新保存
- 任何新增文档，都必须使用规范中文，不允许混用乱码、表情式符号或不可见字符
- 若新增新的脚本类型，需要同步更新 `.editorconfig` 与 `.gitattributes`
