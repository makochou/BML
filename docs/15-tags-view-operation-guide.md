# 标签页批量关闭功能开发说明

## 1. 功能目标

本次改造为后台布局中的标签页区域新增统一的批量关闭能力。

支持的操作如下：

- 关闭当前标签
- 关闭其他标签页
- 关闭所有标签页

本次实现遵循以下约束：

- `affix` 固定标签始终保留，任何入口都不能直接关闭
- `visitedViews` 与 `cachedViews` 必须始终同步
- 当前激活标签被关闭后，跳转目标必须稳定可预测
- 单标签右键菜单与单标签关闭按钮共用同一套动作逻辑

## 2. 交互说明

### 2.1 单标签右键菜单

每个标签都支持右键菜单。

使用方式如下：

1. 在目标标签上点击鼠标右键
2. 选择“关闭当前标签 / 关闭其他标签页 / 关闭所有标签页”
3. 系统以被右键的标签作为动作目标执行处理

这意味着：

- 右键非当前标签并选择“关闭其他标签页”时，会保留该右键标签和所有固定标签
- 如果当前激活页不在保留结果中，系统会自动跳转到右键目标标签

### 2.2 单标签关闭按钮

原有标签上的关闭图标保留。

行为规则如下：

- 点击普通标签的关闭图标，只关闭该标签
- 如果关闭的是当前激活页，则按统一跳转规则选择下一个目标页
- 固定标签不显示关闭图标

## 3. 代码结构说明

### 3.1 组件层

文件：

- `bml-frontend/src/components/TagsView.vue`

组件层主要职责如下：

- 渲染标签列表
- 渲染每个标签的右键菜单
- 统一分发标签动作
- 在关闭动作完成后执行路由跳转

组件内部将两种入口统一收敛到同一个动作执行函数中：

- 标签右键菜单
- 单标签关闭按钮

这样做的好处是后续新增标签操作时，只需要扩展动作枚举、菜单工厂和动作分发逻辑，不需要分别维护多套实现。

### 3.2 Store 层

文件：

- `bml-frontend/src/store/tagsView.ts`

Store 层负责标签状态与缓存状态管理。

保留的核心状态如下：

- `visitedViews`
- `cachedViews`

保留的核心公开方法如下：

- `addView`
- `addVisitedView`
- `addCachedView`
- `delView`
- `delOthersViews`
- `delAllViews`

## 4. 状态与缓存同步机制

标签页系统不仅要管理展示出来的标签，还要维护 `keep-alive` 依赖的缓存列表。

### 4.1 为什么不能只删除标签，不处理缓存

如果只删除 `visitedViews`，却没有同步更新 `cachedViews`，会出现以下问题：

- 页面标签已经不在了，但缓存实例仍然存在
- 保留标签与缓存列表不一致
- 后续重新打开页面时，可能出现旧状态残留

### 4.2 本次采用的处理方式

本次没有继续沿用“从旧缓存数组中裁剪一段”的方式，而是统一改成：

1. 先根据关闭动作得到保留后的 `visitedViews`
2. 再基于保留后的标签重新生成 `cachedViews`

缓存重建时只保留满足以下条件的视图：

- 存在 `name`
- 没有设置 `meta.noCache`
- 该视图仍然存在于保留后的 `visitedViews`

### 4.3 固定标签处理规则

固定标签由 `meta.affix` 标识。

固定标签的规则如下：

- 不允许执行“关闭当前标签”
- 执行“关闭所有标签页”时也必须保留
- 执行“关闭其他标签页”时，固定标签始终保留

## 5. 路由跳转规则

### 5.1 关闭当前标签

如果关闭的是当前激活标签，系统按以下顺序寻找跳转目标：

1. 原标签右侧相邻且仍然保留的标签
2. 原标签左侧相邻且仍然保留的标签
3. 第一个固定标签
4. `/admin/dashboard`

这样可以保证标签关闭后的页面切换更稳定，也更符合常见后台系统的使用预期。

### 5.2 关闭其他标签页

关闭其他标签页后：

- 若当前激活页仍被保留，则页面保持不动
- 若当前激活页已被移除，则跳转到目标标签
- 若目标标签不存在，则回退到固定标签

### 5.3 关闭所有标签页

关闭所有标签页后：

- 只保留固定标签
- 自动跳转到第一个固定标签
- 若极端情况下没有固定标签，则回退到 `/admin/dashboard`

## 6. 如何扩展新的标签动作

如果后续需要增加新的标签动作，例如：

- 刷新当前标签
- 关闭左侧标签
- 关闭右侧标签

建议按以下顺序扩展：

1. 在 `TagsView.vue` 中扩展 `TagActionKey`
2. 为新动作补充展示文案
3. 在 `createActionOptions` 中定义启用和禁用规则
4. 在统一动作分发函数中实现具体处理逻辑
5. 如涉及状态变化，在 `tagsView.ts` 中补充对应的 store 方法
6. 为新能力补充单元测试

## 7. 自动化测试说明

文件：

- `bml-frontend/src/store/tagsView.test.ts`

当前已覆盖的关键场景如下：

- 相同路径标签去重
- 删除普通标签后同步更新缓存
- 固定标签不可删除
- 关闭其他标签页时保留目标标签与固定标签
- 批量关闭后按保留结果重建缓存
- `noCache` 标签不会进入缓存
- 关闭所有标签页后仅保留固定标签
- 固定标签即使保留，也会遵循 `noCache` 规则

## 8. 常见问题与排查

### 8.1 关闭标签后页面跳转不符合预期

优先检查：

- 目标标签是否真的是当前激活标签
- 关闭后的保留标签顺序是否符合预期
- 固定标签是否正确配置了 `meta.affix`

### 8.2 标签已经关闭，但页面状态仍然保留

优先检查：

- 该视图是否声明了 `name`
- 是否错误配置了 `meta.noCache`
- `cachedViews` 是否按保留结果重建

### 8.3 固定标签被误关闭

优先检查：

- 路由 `meta.affix` 是否丢失
- 初始化固定标签时是否成功注入到 `visitedViews`

## 9. 手工验收清单

建议按以下步骤验收：

1. 打开多个普通标签，确认每个标签都支持右键菜单
2. 右键当前标签执行“关闭当前标签”，确认当前页关闭并跳转到相邻标签
3. 右键当前标签执行“关闭其他标签页”，确认只保留当前页和固定标签
4. 右键任意普通标签执行“关闭所有标签页”，确认只保留固定标签
5. 右键非当前标签执行“关闭当前标签”，确认仅关闭该标签
6. 右键非当前标签执行“关闭其他标签页”，确认会切换到该标签
7. 右键固定标签，确认“关闭当前标签”为禁用状态
8. 连续执行多次关闭操作，确认控制台无报错
9. 执行 `npm test`
10. 执行 `npm run build`

## 10. 后续维护建议

- 新增标签动作时，优先复用现有统一动作分发模型
- 不要在组件层直接手工修改 `cachedViews`
- 所有批量关闭能力都应通过 store 重建缓存
- 若将来支持按 `fullPath` 多开标签，需要同步调整标签唯一性判断与缓存策略
