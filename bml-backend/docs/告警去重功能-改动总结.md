### 2. SysAlertServiceImpl.java

**文件路径**：`bml-backend/bml-modules/bml-module-system/src/main/java/com/bml/module/system/service/impl/SysAlertServiceImpl.java`

**改动类型**：新增方法实现

**改动内容**：

```java
/**
 * 保存或更新告警记录（智能去重）
 * <p>
 * 核心逻辑：
 * <ol>
 *   <li>根据 alertType 和 alertTitle 查询是否存在未读的相同告警</li>
 *   <li>如果存在，更新该告警的内容和更新时间（保持未读状态）</li>
 *   <li>如果不存在，创建新的告警记录</li>
 * </ol>
 * 这样可以确保每种类型的许可证变更只显示最新的一条告警，避免重复提醒。
 * </p>
 *
 * @param alert 告警对象（必须包含 alertType、alertTitle、alertContent）
 * @return 保存或更新后的告警对象
 */
@Override
public SysAlert saveOrUpdateAlert(SysAlert alert) {
    // 查询是否存在相同类型和标题的未读告警
    SysAlert existingAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 0)
            .last("LIMIT 1"));

    if (existingAlert != null) {
        // 更新现有告警的内容和级别（保持未读状态，更新时间会自动更新）
        existingAlert.setAlertContent(alert.getAlertContent());
        existingAlert.setAlertLevel(alert.getAlertLevel());
        this.updateById(existingAlert);
        return existingAlert;
    } else {
        // 创建新告警记录
        this.save(alert);
        return alert;
    }
}
```

**改动说明**：

- 实现 `saveOrUpdateAlert` 方法的核心逻辑
- 使用 MyBatis-Plus 的 `LambdaQueryWrapper` 查询现有告警
- 根据查询结果决定更新或创建
- 详细的 JavaDoc 注释说明实现逻辑

### 3. SysLicenseController.java

**文件路径**：`bml-backend/bml-modules/bml-module-system/src/main/java/com/bml/module/system/controller/SysLicenseController.java`

**改动类型**：修改方法实现

**改动内容**：

```java
/**
 * 通用告警保存工具方法（智能去重版本）
 * <p>
 * 统一封装 SysAlert 对象的构建与持久化，避免在 createLicenseChangeAlerts 中重复样板代码。
 * 所有许可证变更告警均通过此方法写入，便于后续统一调整字段默认值。
 * </p>
 * <p>
 * <strong>智能去重机制：</strong>
 * 在保存新告警前，先检查是否已存在相同类型（alertType）和标题（alertTitle）的未读告警。
 * 如果存在，则更新该告警的内容和时间，而不是创建新记录。
 * 这样可以确保每种类型的许可证变更只显示最新的一条告警，避免重复提醒用户。
 * </p>
 *
 * @param alertType    告警类型标识（如 LICENSE_CHANGE）
 * @param alertLevel   告警级别（info / warning / critical）
 * @param alertTitle   告警标题（简短描述变更内容）
 * @param alertContent 告警详情（完整描述变更前后的值）
 */
private void saveAlert(String alertType, String alertLevel, String alertTitle, String alertContent) {
    SysAlert alert = new SysAlert();
    alert.setAlertType(alertType);
    alert.setAlertLevel(alertLevel);
    alert.setAlertTitle(alertTitle);
    alert.setAlertContent(alertContent);
    alert.setReadStatus(0);
    // 使用智能去重保存方法，避免重复告警
    sysAlertService.saveOrUpdateAlert(alert);
}
```

**改动说明**：

- 修改 `saveAlert` 方法，使用新的 `saveOrUpdateAlert` 方法
- 更新 JavaDoc 注释，说明智能去重机制
- 保持方法签名不变，确保向后兼容

## 🗄️ 数据库改动

### V1.9.6__optimize_alert_deduplication.sql

**文件路径**：`bml-backend/bml-app/src/main/resources/db/migration/V1.9.6__optimize_alert_deduplication.sql`

**改动类型**：新增数据库迁移脚本

**改动内容**：

```sql
-- =============================================================================
-- 版本: V1.9.6
-- 描述: 优化告警去重机制
-- 作者: BML Team
-- 日期: 2026-04-30
-- =============================================================================

-- 为 sys_alert 表添加复合索引，优化告警去重查询性能
-- 该索引用于快速查找相同类型和标题的未读告警，支持智能去重功能
-- 索引列顺序：alert_type, alert_title, read_status, deleted
-- 这样可以高效支持 WHERE alert_type = ? AND alert_title = ? AND read_status = 0 AND deleted = 0 的查询
CREATE INDEX idx_alert_dedup ON sys_alert (alert_type, alert_title, read_status, deleted);

-- 说明：
-- 1. 该索引不是唯一索引，因为允许存在多条已读的相同告警（历史记录）
-- 2. 去重逻辑在应用层实现（SysAlertServiceImpl.saveOrUpdateAlert），确保灵活性
-- 3. 索引包含 deleted 字段，因为 MyBatis-Plus 的逻辑删除会自动添加 deleted = 0 条件
```

**改动说明**：

- 创建复合索引 `idx_alert_dedup`，优化查询性能
- 索引列顺序与查询条件匹配，确保最佳性能
- 详细的注释说明索引用途和设计考虑
