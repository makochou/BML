# 告警提示策略优化方案

## 📋 问题分析

### 用户反馈

> "这个一直在提示啊"

### 问题现象

从截图可以看到：
1. 告警中心显示了 20 条告警
2. 所有告警标题都是"API 账号上限升级"
3. 告警内容各不相同，说明用户进行了多次许可证更新
4. 每次更新都会创建/更新告警，导致告警一直显示

### 问题根源

**当前逻辑**：
- 每次更新许可证时，系统都会创建或更新告警
- 如果告警未读，会更新告警内容
- 如果告警已读，会创建新告警
- **结果**：用户每次更新许可证都会看到告警提示

**用户期望**：
- 许可证更新后，只提示一次
- 用户查看并标记已读后，不再重复提示
- 除非有新的变更类型（如从升级变为降级）

## ✅ 解决方案

### 方案一：基于已读状态的智能提示（推荐）

**核心思路**：
- 如果存在相同类型和标题的**已读**告警，且时间在24小时内，则不再创建新告警
- 如果不存在或已超过24小时，则创建新告警
- 这样可以避免短时间内重复提示，同时保留长期的告警记录

**优势**：
- ✅ 避免短时间内重复提示
- ✅ 保留长期的告警记录
- ✅ 用户体验更好

**实现**：

```java
/**
 * 保存或更新告警记录（智能去重 + 防重复提示）
 * <p>
 * 核心逻辑：
 * <ol>
 *   <li>查询是否存在相同类型和标题的未读告警，如果存在则更新</li>
 *   <li>查询是否存在相同类型和标题的已读告警（24小时内），如果存在则不创建新告警</li>
 *   <li>否则创建新告警</li>
 * </ol>
 * </p>
 */
@Override
public SysAlert saveOrUpdateAlert(SysAlert alert) {
    // 1. 查询是否存在相同类型和标题的未读告警
    SysAlert existingUnreadAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 0)
            .last("LIMIT 1"));

    if (existingUnreadAlert != null) {
        // 更新现有未读告警
        existingUnreadAlert.setAlertContent(alert.getAlertContent());
        existingUnreadAlert.setAlertLevel(alert.getAlertLevel());
        this.updateById(existingUnreadAlert);
        return existingUnreadAlert;
    }

    // 2. 查询是否存在相同类型和标题的已读告警（24小时内）
    LocalDateTime oneDayAgo = LocalDateTime.now().minusHours(24);
    SysAlert recentReadAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 1)
            .gt(SysAlert::getUpdateTime, oneDayAgo)
            .orderByDesc(SysAlert::getUpdateTime)
            .last("LIMIT 1"));

    if (recentReadAlert != null) {
        // 24小时内已有已读告警，不再创建新告警
        log.info("[Alert] 24小时内已有已读告警，跳过创建：{}", alert.getAlertTitle());
        return recentReadAlert;
    }

    // 3. 创建新告警
    LocalDateTime now = LocalDateTime.now();
    alert.setCreateTime(now);
    alert.setUpdateTime(now);
    this.save(alert);
    return alert;
}
```

### 方案二：基于内容哈希的去重（备选）

**核心思路**：
- 为每条告警生成内容哈希（基于类型、标题、关键内容）
- 如果存在相同哈希的告警（24小时内），则不创建新告警
- 这样可以避免内容完全相同的告警重复创建

**优势**：
- ✅ 更精确的去重
- ✅ 避免内容完全相同的告警重复

**劣势**：
- ❌ 实现复杂度较高
- ❌ 需要额外的哈希字段

### 方案三：基于用户偏好的配置（长期方案）

**核心思路**：
- 允许用户配置告警提示策略
- 选项：每次提示 / 24小时内只提示一次 / 已读后不再提示
- 存储在用户配置表中

**优势**：
- ✅ 灵活性最高
- ✅ 满足不同用户的需求

**劣势**：
- ❌ 需要额外的配置表和界面
- ❌ 实现周期较长

## 🔧 推荐实现（方案一）

### 1. 修改 SysAlertServiceImpl.java

```java
/**
 * 保存或更新告警记录（智能去重 + 防重复提示）
 * <p>
 * 核心逻辑：
 * <ol>
 *   <li>查询是否存在相同类型和标题的未读告警，如果存在则更新</li>
 *   <li>查询是否存在相同类型和标题的已读告警（24小时内），如果存在则不创建新告警</li>
 *   <li>否则创建新告警</li>
 * </ol>
 * 这样可以避免短时间内重复提示，同时保留长期的告警记录。
 * </p>
 * <p>
 * <strong>防重复提示机制</strong>：
 * 如果用户在24小时内已经查看并标记已读了相同类型的告警，
 * 则不再创建新告警，避免重复打扰用户。
 * </p>
 *
 * @param alert 告警对象（必须包含 alertType、alertTitle、alertContent）
 * @return 保存或更新后的告警对象，如果跳过创建则返回最近的已读告警
 */
@Override
public SysAlert saveOrUpdateAlert(SysAlert alert) {
    // 1. 查询是否存在相同类型和标题的未读告警
    SysAlert existingUnreadAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 0)
            .last("LIMIT 1"));

    if (existingUnreadAlert != null) {
        // 更新现有未读告警的内容和级别（保持未读状态，update_time 会自动更新）
        existingUnreadAlert.setAlertContent(alert.getAlertContent());
        existingUnreadAlert.setAlertLevel(alert.getAlertLevel());
        this.updateById(existingUnreadAlert);
        log.debug("[Alert] 更新现有未读告警：ID={}, 标题={}", existingUnreadAlert.getId(), alert.getAlertTitle());
        return existingUnreadAlert;
    }

    // 2. 查询是否存在相同类型和标题的已读告警（24小时内）
    java.time.LocalDateTime oneDayAgo = java.time.LocalDateTime.now().minusHours(24);
    SysAlert recentReadAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 1)
            .gt(SysAlert::getUpdateTime, oneDayAgo)
            .orderByDesc(SysAlert::getUpdateTime)
            .last("LIMIT 1"));

    if (recentReadAlert != null) {
        // 24小时内已有已读告警，不再创建新告警，避免重复打扰用户
        log.info("[Alert] 24小时内已有已读告警，跳过创建：类型={}, 标题={}, 上次已读时间={}",
                alert.getAlertType(), alert.getAlertTitle(), recentReadAlert.getUpdateTime());
        return recentReadAlert;
    }

    // 3. 创建新告警记录，设置初始的 update_time
    java.time.LocalDateTime now = java.time.LocalDateTime.now();
    alert.setCreateTime(now);
    alert.setUpdateTime(now);
    this.save(alert);
    log.info("[Alert] 创建新告警：ID={}, 类型={}, 标题={}", alert.getId(), alert.getAlertType(), alert.getAlertTitle());
    return alert;
}
```

### 2. 添加配置项（可选）

如果希望24小时的时间窗口可配置，可以添加配置项：

```yaml
# application.yml
bml:
  alert:
    # 告警防重复提示时间窗口（小时），默认24小时
    # 在此时间窗口内，相同类型和标题的已读告警不会重复创建
    duplicate-prevention-hours: 24
```

```java
@Component
@ConfigurationProperties(prefix = "bml.alert")
@Data
public class AlertProperties {
    /**
     * 告警防重复提示时间窗口（小时）
     * 在此时间窗口内，相同类型和标题的已读告警不会重复创建
     */
    private int duplicatePreventionHours = 24;
}
```

**优化后**：
- 告警中心更加整洁
- 只保留必要的告警记录
- 用户能够快速找到重要信息

## 🔍 常见问题

### Q1：为什么选择24小时作为时间窗口？

**A**：24小时是一个合理的平衡点：
- 太短（如1小时）：可能无法有效防止重复提示
- 太长（如7天）：可能导致用户错过重要变更
- 24小时：既能防止频繁打扰，又能保证重要信息及时送达

### Q2：如果用户希望每次都提示怎么办？

**A**：可以通过配置项调整时间窗口：
- 设置为 0：每次都提示（恢复原有行为）
- 设置为 24：24小时内不重复提示（推荐）
- 设置为 168：7天内不重复提示

### Q3：这会影响其他类型的告警吗？

**A**：不会。防重复提示逻辑基于告警类型和标题，不同类型的告警互不影响。

### Q4：如果用户没有标记已读会怎样？

**A**：如果用户没有标记已读，告警会一直保持未读状态，并且会被更新（而不是创建新告警）。这样用户始终能看到最新的变更信息。
