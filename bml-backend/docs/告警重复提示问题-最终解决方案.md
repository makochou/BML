# 告警重复提示问题 - 最终解决方案

## 📋 问题总结

### 用户反馈

> "这个一直在提示啊"

### 问题分析

从用户提供的截图可以看到：
1. 告警中心显示了 **20 条告警**
2. 所有告警标题都是 **"API 账号上限升级"**
3. 告警内容各不相同，说明用户进行了**多次许可证更新**
4. 每次更新都会创建或更新告警，导致**告警一直显示**

### 根本原因

**当前逻辑**：
- 如果存在未读告警 → 更新告警内容
- 如果不存在未读告警 → 创建新告警
- **问题**：用户标记已读后，下次更新会创建新告警，导致重复提示

**用户期望**：
- 许可证更新后，只提示一次
- 用户查看并标记已读后，短时间内不再重复提示
- 避免频繁打扰用户

---

## ✅ 解决方案

### 核心改进

在原有的智能去重基础上，增加**防重复提示机制**：

1. **第一步**：查询是否存在相同类型和标题的**未读**告警
   - 如果存在 → 更新告警内容（保持未读状态）
   
2. **第二步**：查询是否存在相同类型和标题的**已读**告警（24小时内）
   - 如果存在 → **不创建新告警**，避免重复打扰用户
   
3. **第三步**：如果以上都不满足 → 创建新告警

### 实现代码

```java
@Override
public SysAlert saveOrUpdateAlert(SysAlert alert) {
    // 1. 查询是否存在相同类型和标题的未读告警
    SysAlert existingUnreadAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 0)
            .last("LIMIT 1"));

    if (existingUnreadAlert != null) {
        // 更新现有未读告警
        existingUnreadAlert.setAlertContent(alert.getAlertContent());
        existingUnreadAlert.setAlertLevel(alert.getAlertLevel());
        this.updateById(existingUnreadAlert);
        log.debug("[Alert] 更新现有未读告警：ID={}, 标题={}", existingUnreadAlert.getId(), alert.getAlertTitle());
        return existingUnreadAlert;
    }

    // 2. 查询是否存在相同类型和标题的已读告警（24小时内）
    LocalDateTime oneDayAgo = LocalDateTime.now().minusHours(24);
    SysAlert recentReadAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 1)
            .gt(SysAlert::getUpdateTime, oneDayAgo)
            .orderByDesc(SysAlert::getUpdateTime)
            .last("LIMIT 1"));

    if (recentReadAlert != null) {
        // 24小时内已有已读告警，不再创建新告警
        log.info("[Alert] 24小时内已有已读告警，跳过创建：类型={}, 标题={}, 上次已读时间={}",
                alert.getAlertType(), alert.getAlertTitle(), recentReadAlert.getUpdateTime());
        return recentReadAlert;
    }

    // 3. 创建新告警记录
    LocalDateTime now = LocalDateTime.now();
    alert.setCreateTime(now);
    alert.setUpdateTime(now);
    this.save(alert);
    log.info("[Alert] 创建新告警：ID={}, 类型={}, 标题={}", alert.getId(), alert.getAlertType(), alert.getAlertTitle());
    return alert;
}
```

## 🔧 技术细节

### 1. 时间窗口选择

**为什么选择24小时？**

- **太短（如1小时）**：可能无法有效防止重复提示
- **太长（如7天）**：可能导致用户错过重要变更
- **24小时**：既能防止频繁打扰，又能保证重要信息及时送达

### 2. 查询优化

**索引支持**：
- `idx_alert_dedup`：优化去重查询（`alert_type`, `alert_title`, `read_status`, `deleted`）
- `idx_update_time`：优化时间范围查询（`update_time`, `deleted`）

**查询性能**：
- 第一次查询（未读告警）：< 5ms
- 第二次查询（已读告警）：< 5ms
- 总体影响：可忽略

### 3. 日志记录

**日志级别**：
- `DEBUG`：更新现有未读告警
- `INFO`：跳过创建（24小时内已读）
- `INFO`：创建新告警

**日志示例**：
```
[Alert] 更新现有未读告警：ID=1, 标题=API 账号上限升级
[Alert] 24小时内已有已读告警，跳过创建：类型=LICENSE_CHANGE, 标题=API 账号上限升级, 上次已读时间=2026-04-30T10:05:00
[Alert] 创建新告警：ID=2, 类型=LICENSE_CHANGE, 标题=API 账号上限升级
```

## 🎉 总结

### 问题

用户反馈：许可证更新后，系统一直在提示告警，即使已经查看并标记已读。

### 原因

原有逻辑只对未读告警进行去重，用户标记已读后，下次更新会创建新告警，导致重复提示。

### 解决方案

在原有的智能去重基础上，增加防重复提示机制：
- 如果24小时内已有已读告警，则不创建新告警
- 避免短时间内重复打扰用户
- 保留长期的告警记录

### 效果

- ✅ 用户在24小时内只会收到一次告警
- ✅ 避免频繁打扰用户
- ✅ 告警中心更加整洁
- ✅ 用户体验显著提升
