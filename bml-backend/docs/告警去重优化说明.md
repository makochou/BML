# 告警去重优化说明

## 问题描述

在原有实现中，每次更新许可证时，系统都会为每个变更维度（如"API 账号上限升级"、"业务用户上限升级"等）创建新的告警记录。这导致以下问题：

1. **重复告警**：即使是相同类型的变更，每次更新都会产生新的告警记录
2. **告警泛滥**：用户在告警中心看到大量重复的告警信息，影响用户体验
3. **信息冗余**：历史告警记录中存在大量相似内容，增加数据库存储压力

### 问题场景示例

当管理员将 API 账号配额从 5 个升级到 10 个时，系统会创建一条告警：
```
标题：API 账号上限升级
内容：许可证更新：最大 API 账号数从 5 变更为 10。
```

如果管理员再次将配额从 10 个升级到 15 个，系统会再次创建一条新告警：
```
标题：API 账号上限升级
内容：许可证更新：最大 API 账号数从 10 变更为 15。
```

这样用户在告警中心会看到两条相同标题的告警，造成信息冗余。

## 解决方案

### 核心思路

实现**智能去重机制**：在保存新告警前，先检查是否已存在相同类型（`alertType`）和标题（`alertTitle`）的未读告警。如果存在，则更新该告警的内容和时间，而不是创建新记录。

### 实现细节

#### 1. 新增服务接口方法

在 `ISysAlertService` 接口中新增 `saveOrUpdateAlert` 方法：

```java
/**
 * 保存或更新告警记录（智能去重）
 * <p>
 * 在保存新告警前，先检查是否已存在相同类型和标题的未读告警。
 * 如果存在，则更新该告警的内容和时间；否则创建新记录。
 * 这样可以避免重复告警，确保每种类型的变更只显示最新的一条。
 * </p>
 *
 * @param alert 告警对象
 * @return 保存或更新后的告警对象
 */
SysAlert saveOrUpdateAlert(SysAlert alert);
```

#### 2. 实现智能去重逻辑

在 `SysAlertServiceImpl` 中实现去重逻辑：

```java
@Override
public SysAlert saveOrUpdateAlert(SysAlert alert) {
    // 查询是否存在相同类型和标题的未读告警
    SysAlert existingAlert = this.getOne(new LambdaQueryWrapper<SysAlert>()
            .eq(SysAlert::getAlertType, alert.getAlertType())
            .eq(SysAlert::getAlertTitle, alert.getAlertTitle())
            .eq(SysAlert::getReadStatus, 0)
            .last("LIMIT 1"));

    if (existingAlert != null) {
        // 更新现有告警的内容和级别（保持未读状态，更新时间会自动更新）
        existingAlert.setAlertContent(alert.getAlertContent());
        existingAlert.setAlertLevel(alert.getAlertLevel());
        this.updateById(existingAlert);
        return existingAlert;
    } else {
        // 创建新告警记录
        this.save(alert);
        return alert;
    }
}
```

#### 3. 修改告警保存方法

在 `SysLicenseController` 中修改 `saveAlert` 方法，使用新的智能去重方法：

```java
private void saveAlert(String alertType, String alertLevel, String alertTitle, String alertContent) {
    SysAlert alert = new SysAlert();
    alert.setAlertType(alertType);
    alert.setAlertLevel(alertLevel);
    alert.setAlertTitle(alertTitle);
    alert.setAlertContent(alertContent);
    alert.setReadStatus(0);
    // 使用智能去重保存方法，避免重复告警
    sysAlertService.saveOrUpdateAlert(alert);
}
```

## 技术特性

### 1. 去重规则

- **去重维度**：`alertType` + `alertTitle` + `readStatus = 0`
- **去重范围**：仅对未读告警进行去重
- **更新策略**：更新内容（`alertContent`）和级别（`alertLevel`），保持未读状态

### 2. 不去重的场景

以下场景会创建新的告警记录，而不是更新现有记录：

1. **已读告警**：如果相同类型和标题的告警已被用户标记为已读，则创建新记录
2. **不同标题**：即使类型相同，标题不同的告警会分别创建
3. **不同类型**：即使标题相同，类型不同的告警会分别创建

### 3. 性能优化

- **索引支持**：通过复合索引 `idx_alert_dedup` 优化查询性能
- **限制查询**：使用 `LIMIT 1` 限制查询结果，提高查询效率
- **自动更新时间**：利用数据库的 `ON UPDATE CURRENT_TIMESTAMP` 自动更新 `update_time`

## 最佳实践

### 1. 告警标题设计

告警标题应具有唯一性和描述性，建议格式：

```
[资源类型] + [操作类型]
```

示例：
- ✅ "API 账号上限升级"
- ✅ "业务用户上限降级"
- ✅ "许可证有效期延期"
- ❌ "配额变更"（过于笼统）

### 2. 告警内容设计

告警内容应包含完整的变更信息，建议格式：

```
[操作描述]：[资源名称] 从 [旧值] 变更为 [新值]。
```

示例：
```
许可证更新：最大 API 账号数从 5 变更为 10。
```

### 3. 告警级别选择

- `info`：提示性信息（如配额升级）
- `warning`：警告信息（如配额降级）
- `error`：错误信息（如许可证过期）
- `critical`：严重错误（如系统资源耗尽）

## 常见问题

### Q1：为什么只对未读告警去重？

**A**：已读告警代表用户已经知晓该信息，应保留为历史记录。如果对已读告警也去重，会导致历史记录丢失。

### Q2：如果用户在更新期间标记告警为已读会怎样？

**A**：如果用户在两次许可证更新之间标记告警为已读，下次更新时会创建新的告警记录，这是符合预期的行为。

### Q3：去重逻辑会影响其他类型的告警吗？

**A**：不会。去重逻辑基于 `alertType` 和 `alertTitle` 进行匹配，不同类型的告警（如 CPU 告警、磁盘告警）不会相互影响。

### Q4：如何查看历史告警记录？

**A**：所有告警记录（包括已读和未读）都保留在数据库中，可以通过告警中心的历史记录功能查看。

对于短时间内的多次变更，可以考虑合并为一条告警，显示变更历史：

```

- 2026-04-30 10:00: 5 → 10
```

对于长时间未读的告警，可以考虑自动标记为已读或归档，避免告警堆积。

### 3. 告警优先级

引入告警优先级机制，重要告警优先展示，提升用户体验。

支持多种通知渠道（邮件、短信、企业微信等），确保重要告警及时送达。
