# 告警去重功能使用指南

## 功能概述

告警去重功能是针对许可证更新时重复告警问题的优化方案。通过智能去重机制，确保每种类型的许可证变更只显示最新的一条告警，避免重复提醒用户。

## 问题背景

### 原有问题

在原有实现中，每次更新许可证时，系统都会为每个变更维度创建新的告警记录。例如：

1. **第一次更新**：API 账号配额从 5 个升级到 10 个
   - 系统创建告警：「API 账号上限升级：从 5 变更为 10」

2. **第二次更新**：API 账号配额从 10 个升级到 15 个
   - 系统再次创建告警：「API 账号上限升级：从 10 变更为 15」

3. **第三次更新**：API 账号配额从 15 个升级到 20 个
   - 系统再次创建告警：「API 账号上限升级：从 15 变更为 20」

**结果**：用户在告警中心看到 3 条相同标题的告警，造成信息冗余和困扰。

### 用户痛点

- ❌ 告警中心堆积大量重复告警
- ❌ 难以快速找到最新的变更信息
- ❌ 影响用户体验，降低告警的有效性
- ❌ 增加数据库存储压力

## 解决方案

### 智能去重机制

系统在保存新告警前，会自动检查是否已存在相同类型和标题的未读告警：

- **如果存在**：更新该告警的内容和时间，而不是创建新记录
- **如果不存在**：创建新的告警记录

### 优化效果

使用智能去重后，同样的场景会变成：

1. **第一次更新**：API 账号配额从 5 个升级到 10 个
   - 系统创建告警（ID=1）：「API 账号上限升级：从 5 变更为 10」

2. **第二次更新**：API 账号配额从 10 个升级到 15 个
   - 系统更新告警（ID=1）：「API 账号上限升级：从 10 变更为 15」

3. **第三次更新**：API 账号配额从 15 个升级到 20 个
   - 系统更新告警（ID=1）：「API 账号上限升级：从 15 变更为 20」

**结果**：用户在告警中心只看到 1 条告警，显示最新的变更信息。

## 使用场景

### 场景 1：许可证配额升级

**操作步骤**：

1. 管理员上传新的许可证文件，API 账号配额从 5 个升级到 10 个
2. 系统自动创建告警：「API 账号上限升级」
3. 用户在告警中心看到该告警（未读状态）

**后续操作**：

- 如果用户**未读**该告警，再次升级配额时，系统会更新该告警的内容
- 如果用户**已读**该告警，再次升级配额时，系统会创建新的告警

### 场景 2：许可证配额降级

**操作步骤**：

1. 管理员上传新的许可证文件，API 账号配额从 10 个降级到 5 个
2. 系统自动创建告警：「API 账号上限降级」
3. 系统自动停用超额的 API 账号
4. 用户在告警中心看到该告警（未读状态）

**后续操作**：

- 用户点击告警，查看详细信息
- 用户前往授权管理页面，确认被停用的 API 账号
- 用户标记告警为已读

### 场景 3：多次更新许可证

**操作步骤**：

1. 管理员第一次更新许可证，API 账号配额从 5 个升级到 10 个
2. 系统创建告警（ID=1）：「API 账号上限升级：从 5 变更为 10」
3. 用户看到告警但未标记为已读
4. 管理员第二次更新许可证，API 账号配额从 10 个升级到 15 个
5. 系统更新告警（ID=1）：「API 账号上限升级：从 10 变更为 15」
6. 用户在告警中心只看到 1 条告警，显示最新的变更信息

**优势**：

- ✅ 避免重复告警
- ✅ 始终显示最新的变更信息
- ✅ 减少用户的认知负担

## 告警类型说明

### 许可证变更告警

所有许可证变更告警统一使用 `LICENSE_CHANGE` 类型，通过标题区分具体的变更内容：

| 标题 | 说明 | 级别 |
|------|------|------|
| API 账号上限升级 | API 账号配额增加 | info |
| API 账号上限降级 | API 账号配额减少 | info |
| 业务用户上限升级 | 业务用户配额增加 | info |
| 业务用户上限降级 | 业务用户配额减少 | info |
| API 来源用户上限升级 | API 创建用户配额增加 | info |
| API 来源用户上限降级 | API 创建用户配额减少 | info |
| 许可证有效期延期 | 许可证到期日延后 | info |
| 许可证有效期缩短 | 许可证到期日提前 | info |
| 授权功能模块新增 | 新增授权功能模块 | info |
| 授权功能模块移除 | 移除授权功能模块 | info |
| 新许可证已过期 | 上传的许可证已过期 | info |

### 去重规则

- **去重维度**：告警类型（`alertType`）+ 告警标题（`alertTitle`）
- **去重范围**：仅对未读告警进行去重
- **更新策略**：更新内容和级别，保持未读状态

## 用户操作指南

### 查看告警

1. 点击右上角的告警图标（铃铛）
2. 告警面板会显示最近的告警列表
3. 未读告警会以高亮显示

### 标记已读

**单条标记**：

1. 在告警列表中找到要标记的告警
2. 点击「标记已读」按钮
3. 该告警会变为已读状态（灰色显示）

**批量标记**：

1. 点击告警面板底部的「全部已读」按钮
2. 所有未读告警会变为已读状态

### 删除告警

1. 在告警列表中找到要删除的告警
2. 点击「删除」按钮
3. 该告警会从列表中移除（逻辑删除）

### 查看历史告警

1. 点击告警面板顶部的「历史记录」标签
2. 选择日期查看当天的所有告警
3. 可以按告警级别筛选

## 常见问题

### Q1：为什么我看到的告警内容和之前不一样了？

**A**：这是正常现象。如果您在两次许可证更新之间没有标记告警为已读，系统会自动更新告警内容为最新的变更信息。这样可以确保您始终看到最新的配额变化。

### Q2：如果我想查看历史的变更记录怎么办？

**A**：建议在每次查看告警后及时标记为已读。已读的告警会保留在历史记录中，您可以通过告警中心的「历史记录」功能查看。

### Q3：为什么有些告警会重复出现？

**A**：只有未读的告警会被去重。如果您已经标记告警为已读，下次相同类型的变更会创建新的告警记录。这是为了确保您不会错过重要的变更信息。

### Q4：去重功能会影响其他类型的告警吗？

**A**：不会。去重功能只针对相同类型和标题的告警。不同类型的告警（如 CPU 告警、磁盘告警）不会相互影响。

### Q5：如何关闭告警通知？

**A**：目前不支持关闭告警通知。告警是系统的重要功能，用于及时通知您许可证配额的变化。建议您及时查看并标记已读，以保持告警中心的整洁。

## 技术细节

### 数据库变更

系统会自动执行数据库迁移脚本，为 `sys_alert` 表添加复合索引：

```sql
CREATE INDEX idx_alert_dedup ON sys_alert (alert_type, alert_title, read_status, deleted);
```

该索引用于优化告警去重查询性能，无需手动干预。

### 性能影响

- **查询性能**：每次保存告警前需查询一次数据库，但由于有索引支持，性能影响可忽略
- **存储空间**：显著减少重复记录，节省存储空间
- **用户体验**：告警中心更加整洁，用户体验显著提升

### 兼容性

- ✅ 向后兼容，不影响现有告警数据
- ✅ 不影响现有告警查询接口
- ✅ 不影响前端告警展示逻辑

## 最佳实践

### 1. 及时查看告警

建议您定期查看告警中心，及时了解系统的变更信息。

### 2. 标记已读

查看告警后，建议及时标记为已读，以便下次变更时能够创建新的告警记录。

### 3. 定期清理

建议定期清理历史告警，保持告警中心的整洁。可以使用「删除」功能移除不再需要的告警。

### 4. 关注重要告警

对于配额降级、许可证过期等重要告警，建议及时处理，避免影响系统正常使用。
