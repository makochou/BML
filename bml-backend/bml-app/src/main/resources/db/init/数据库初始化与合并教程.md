# BML 数据库初始化与脚本合并教程

> 适用项目：BML 企业中台管理系统（`bml-backend`）
> 数据库：MariaDB 10.4+ / MySQL 8.0+　字符集：`utf8mb4` / `utf8mb4_unicode_ci`
> 文档版本：对应迁移脚本 `V1.0.0` ~ `V2.24.0`（共 44 个）合并后的最终态

---

## 1. 这份教程解决什么问题

后端服务在启动时由 **Flyway** 自动执行 `classpath:db/migration` 下的增量 SQL 脚本，目前已经累积到 **44 个**（`V1.0.0` 到 `V2.24.0`）。这带来两个现实问题：

1. **没有"建库"步骤。** Flyway 只能连接到一个**已经存在**的数据库再做迁移；`application.yml` 里的连接串直接指向 `bml_system` 库。也就是说，全新环境必须先有人手工 `CREATE DATABASE bml_system`，否则后端启动会因为连不上库而失败。
2. **44 个脚本对新人不友好。** 全新部署要顺序跑完几十个增量脚本，其中还有"先建后删""反复修菜单"等历史包袱，既慢又难读。

为此本次产出了**两个等价的合并脚本**和**这份教程**，把"建库 + 建表 + 基础数据"一次性搞定。

### 产物清单

| 文件 | 作用 | 是否含建库/建账号 |
| --- | --- | --- |
| `bml_full_init.sql` | **独立全量初始化脚本**，开箱即用，手工执行一次即可建好整个库 | ✅ 含 |
| `bml_baseline.sql` | **Flyway 合并基线脚本**，作为新的 `V1.0.0` 候选，供 Flyway 连接到已存在的库后执行 | ❌ 不含 |
| `数据库初始化与合并教程.md` | 本文档 | — |

> 两个 SQL 文件的**表结构与基础数据完全相同**，唯一区别是 `bml_full_init.sql` 比 `bml_baseline.sql` 在开头多了"创建数据库 + 创建账号 + 切库"三段。

### 这两个脚本是怎么生成的（可信度说明）

不是手工逐条合并（那样极易出错），而是：把 44 个迁移脚本按 Flyway 的**语义版本顺序**在一套真实的 **MariaDB 10.6** 上**完整跑了一遍**，再从"跑完后的真库"导出最终结构与数据。随后又把合并脚本在**全新空库**上重新执行，与"迁移真库"做了**逐表结构 + 逐行数据**比对：

- ✅ 29 张表结构逐字节一致；
- ✅ 12 张含数据的表、共 629 行数据逐行一致；
- ✅ 执行全程 0 报错。

所以可以放心认为：**执行合并脚本 ≡ 顺序跑完全部 44 个迁移**。

## 2. 环境要求

- 数据库：MariaDB 10.4 及以上，或 MySQL 8.0 及以上。
  - 注意：原迁移脚本 `V1.9.1`、`V1.9.2` 用到了 MariaDB 专有语法 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`，**在 MySQL 8.0 上会报语法错误**。但本次产出的两个**合并脚本不含该语法**（已是最终态的 `CREATE TABLE`），因此在 MySQL 8.0 与 MariaDB 上都能正常执行。生产用的是 MariaDB（驱动 `org.mariadb.jdbc.Driver`），建议本地也用 MariaDB 保持一致。
- 字符集：库、表、连接统一 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`（脚本已固定，勿改成 `utf8`，否则中文/emoji 会乱码或截断）。
- 执行账号：需要有 `CREATE DATABASE`、`CREATE USER`、`GRANT` 权限（通常是 `root`）。

## 3. 方式一：用 `bml_full_init.sql` 全新初始化（最常用）

适用场景：**本地开发首次搭建、新测试环境、全新生产环境**。

### 3.1 第一步：按需修改"配置区"

打开 `bml_full_init.sql`，顶部有一段醒目的 **【配置区】**：

```sql
-- 创建数据库（utf8mb4 完整 Unicode）
CREATE DATABASE IF NOT EXISTS `bml_system`
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

-- 创建业务账号并授权（仅授予该库权限，最小权限原则）
CREATE USER IF NOT EXISTS 'bml'@'%' IDENTIFIED BY 'Bml@2026_ChangeMe';
GRANT ALL PRIVILEGES ON `bml_system`.* TO 'bml'@'%';
FLUSH PRIVILEGES;

USE `bml_system`;
```

需要改什么：

- **库名**：默认 `bml_system`。要改名的话，把整份文件里所有 `bml_system` 全局替换为新库名即可（出现在 `CREATE DATABASE`、`GRANT`、`USE` 三处）。
- **账号 / 密码**：生产环境务必把 `Bml@2026_ChangeMe` 改成强密码；`'bml'@'%'` 中的 `%` 表示允许任意主机连接，若只在本机访问可改成 `'localhost'`。
- **已有账号**：若 DBA 已统一建好账号，可把 `CREATE USER` 和 `GRANT` 两行注释掉。

> 这些值要与后端 `application.yml` 的环境变量对应（见 3.4）。

### 3.2 第二步：执行脚本

任选一种方式：

**命令行（推荐）**

```bash
# MySQL
mysql  -u root -p < bml_full_init.sql
# MariaDB
mariadb -u root -p < bml_full_init.sql
```

**已登录的客户端里用 `source`**

```sql
mysql -u root -p
mysql> SOURCE /绝对路径/bml_full_init.sql;
```

**图形化工具**：Navicat / DBeaver / DataGrip 直接打开该 `.sql` 文件 → 选中全部 → 执行即可。

### 3.3 第三步：校验结果

```sql
USE bml_system;
SHOW TABLES;                          -- 应有 29 张表
SELECT COUNT(*) FROM sys_menu;        -- 应为 285
SELECT username, nickname FROM sys_user;  -- 应有 admin / 超级管理员
```

默认超级管理员账号：

- 用户名：`admin`
- 密码：`manager`（BCrypt 存储，密文为 `$2a$10$nL/...`）
- **首次登录后请立即修改密码！**

> 另外，中台管理平台（`/admin`）的账号 **不在数据库里**，由 `application.yml` 的 `bml.admin`（默认 `bml` / `bml123`）配置，与上面的业务账号 `admin` 是两套，互不影响。

### 3.4 第四步：让后端连上这个库

后端 `application.yml` 的数据源是通过环境变量注入的，初始化完成后保证下列变量与脚本一致即可：

```yaml
spring:
  datasource:
    url:      ${BML_DB_URL:jdbc:mariadb://localhost:3306/bml_system?...}
    username: ${BML_DB_USERNAME:root}
    password: ${BML_DB_PASSWORD:sa@123}
```

例如使用脚本里创建的业务账号：

```bash
export BML_DB_URL="jdbc:mariadb://localhost:3306/bml_system?useUnicode=true&characterEncoding=utf-8&useSSL=false&serverTimezone=Asia/Shanghai"
export BML_DB_USERNAME="bml"
export BML_DB_PASSWORD="Bml@2026_ChangeMe"
```

## 4. Flyway 是怎么工作的（理解这一节，后面就不踩坑）

后端启动时，Spring Boot 会自动拉起 Flyway 执行迁移。关键点：

- **脚本位置**：`classpath:db/migration`，命名规范 `V{版本}__{描述}.sql`（两个下划线）。
- **执行顺序**：按**语义版本号**从小到大，不是文件名字典序。所以 `V2.9.0` 在 `V2.10.0` **之前**执行（9 < 10），`V2.15.0` 在 `V2.15.1` 之前。
- **执行记录**：Flyway 在库里维护一张 `flyway_schema_history` 表，记录每个脚本的版本、校验码（checksum）、执行结果。已经记录过的脚本**不会重复执行**（幂等）。
- **本项目的两处定制**（位于 `com.bml.app.config`）：
  - `FlywayRepairConfig`：迁移前先 `repair()` 再 `migrate()`，自动修复因换行符（CRLF/LF）或文件微调导致的 **Checksum mismatch**；
  - `FlywayChineseLogInterceptor`：迁移各阶段输出中文日志。
- **`application.yml` 里的相关配置**：

  ```yaml
  spring:
    flyway:
      enabled: true                # 启用
      locations: classpath:db/migration
      baseline-on-migrate: true    # 允许在"已有内容"的库上首次建立基线
      baseline-version: 0          # 基线版本，低于它的脚本不执行
      out-of-order: true           # 允许乱序补跑旧版本脚本（开发期方便，生产建议关）
  ```

记住一个核心事实：**Flyway 不会建库，只会在已存在的库里建表/改表。** 这就是为什么需要 `bml_full_init.sql` 来补上"建库"这一步。

## 5. 方式二：把合并脚本作为 Flyway 新基线

目标是让**新环境不必再跑 44 个增量脚本**，而是从 `bml_baseline.sql` 这个"合并基线"起步。下面给出三种方案，按你的处境选一种。

> ✅ 当前状态：仓库已采用 **方案 A**。原有 44 个迁移脚本已归档到 `db/migration_archive/`（仅留作历史参考，不再执行）；`db/migration/` 下现在只有合并基线 `V1.0.0__init_schema.sql`。下面三种方案中，方案 A 即当前生效方案，方案 B/C 供其它处境参考。

### 方案 A —— 全新仓库 / 全新项目（推荐，最干净）

适合：还没有任何线上库，或可以随时重建库的早期阶段。

1. 把现有 44 个脚本移到仓库外或归档目录（**移出** `db/migration`，例如新建 `db/migration_archive/` 存放，留作历史参考）。
2. 将 `bml_baseline.sql` 复制为新的基线：
   ```
   db/migration/V1.0.0__init_schema.sql   ← 内容用 bml_baseline.sql
   ```
3. `application.yml` 保持 `baseline-version: 0`、`baseline-on-migrate: true` 即可。
4. 以后所有新变更从 `V2.25.0__xxx.sql` 继续往后追加（版本号不要回退）。

结果：全新空库启动后，Flyway 只执行这一个基线脚本就建好了全部表与数据，干净又快。

### 方案 B —— 已有正在运行的库，想切换到合并基线

适合：已有开发/测试库跑过旧的 44 个脚本，现在想精简脚本目录。

风险点：旧库的 `flyway_schema_history` 里记录了 `V1.0.0`~`V2.24.0`。如果直接用"新的单个 `V1.0.0`"替换，新脚本的 checksum 与历史记录对不上，Flyway 校验会失败。安全做法：

1. 备份数据库（务必）。
2. 按方案 A 完成脚本目录替换（归档旧脚本、放入新基线 `V1.0.0`）。
3. 对每个已有库，清理历史记录并重建基线（二选一）：
   - 用 Flyway 命令行：`flyway repair` 之后 `flyway baseline -baselineVersion=1.0.0`；
   - 或手工把 `flyway_schema_history` 表里旧的 `V1.1.0`~`V2.24.0` 记录删掉，只保留/重置 `V1.0.0` 一条基线记录。
4. 启动后端验证：Flyway 应判定"无需迁移"，应用正常起来。

> 本项目已内置 `repair()`（见第 4 节），多数 checksum 问题会被自动修复；但跨基线重置仍建议手工确认 `flyway_schema_history`。

### 方案 C —— 不动现状，仅补"建库"能力（零风险）

适合：现有环境一切正常，只想解决"全新环境没人建库"的问题，不想碰 Flyway 历史。

做法：**保留全部 44 个迁移脚本不变**，新环境的部署流程改成两步：

1. 先用 `bml_full_init.sql` 建好库（此时表与数据已是最终态）；
2. 后端启动时 Flyway 连上这个库——因为表已存在且都用 `CREATE TABLE IF NOT EXISTS`，配合 `baseline-on-migrate: true`，Flyway 会以当前为基线，不会破坏数据。

> 若希望 Flyway 在这种"已初始化库"上**完全不再补跑历史脚本**，可把 `baseline-version` 设为最新版本（如 `2.24`），这样它只会执行 `V2.25.0+` 的新脚本。注意该配置是全局的，调整前先确认所有环境的初始化方式一致。

## 7. 内置基础数据说明

执行脚本后，库里会预置以下"开箱即用"数据（共 12 张表有数据）：

- **超级管理员**：`sys_user` 1 条（`admin` / `manager`）、`sys_role` 1 条（超级管理员 `admin`，全部权限）、`sys_user_role` 绑定关系 1 条。
- **组织架构样例**：`sys_org` 3 条、`sys_dept` 26 条、`sys_post` 8 条（供演示与快速上手）。
- **菜单与授权**：`sys_menu` 285 条，顶级菜单为「工作台 / 资产目录 / 授权治理 / 系统管理 / 系统监控」；`sys_role_menu` 280 条（超级管理员的菜单授权）。
- **数据字典**：`sys_dict_type` 3 类（`sys_common_status` 系统状态、`sys_user_gender` 用户性别、`sys_menu_type` 菜单类型），`sys_dict_data` 9 条。
- **系统参数**：`sys_config` 11 条（登录页品牌/背景/验证码、空闲超时、审计归档策略等 `sys.login.*`、`sys.audit.*`、`sys.sidebar.*`）。
- **内置定时任务**：`sys_job` 1 条（服务器资源巡检，默认暂停）。

`sys_field_permission` 等其余 17 张表初始为空，属正常最终态（由业务运行时写入）。

## 8. 常见问题 FAQ

**Q1：中文 / emoji 显示成乱码或 `????`。**
连接没用 utf8mb4。命令行加 `--default-character-set=utf8mb4`，连接串带 `characterEncoding=utf-8`，并确认库/表都是 `utf8mb4`（脚本已固定，别改成 `utf8`）。

**Q2：执行报 `You have an error in your SQL syntax ... near 'IF NOT EXISTS'`。**
你在 **MySQL** 上跑了**原始迁移脚本**里的 `ALTER ... ADD COLUMN IF NOT EXISTS`（MariaDB 专有语法）。解决：用本次的**合并脚本**（不含该语法），或换用 MariaDB。

**Q3：第二次执行 `bml_full_init.sql` 报主键冲突 / Duplicate entry。**
基础数据是普通 `INSERT`，**只能在空库执行一次**。需要重置请先 `DROP DATABASE bml_system;` 再重新执行（注意会清空数据）。建表语句用了 `IF NOT EXISTS`，重复执行不会报错，但数据 `INSERT` 会。

**Q4：后端启动报 `Unknown database 'bml_system'`。**
还没建库。Flyway 不会建库，先执行 `bml_full_init.sql`（方式一）。

**Q5：Flyway 报 `Migration checksum mismatch`。**
通常是脚本被编辑过或换行符变化。本项目启动时会自动 `repair()` 修复；若仍失败，手工执行 `flyway repair` 或核对 `flyway_schema_history`。

**Q6：`Access denied` / 账号没权限。**
执行建库/建账号需要高权限账号（如 `root`）；业务账号 `bml` 只对 `bml_system` 库有权限，不能用它来建库。

**Q7：时间相关字段差几个小时。**
连接串设置 `serverTimezone=Asia/Shanghai`，并确保数据库服务器时区正确。

## 9. 后续开发规范建议

- **新增表 / 改字段**：不要修改已发布的历史脚本，新建 `V2.25.0__xxx.sql` 增量脚本（版本号只增不减）；表与字段务必带中文 `COMMENT`。
- **公共字段**：新表沿用 `id（雪花）/ create_by / create_time / update_by / update_time / deleted / version` 七件套，与现有表保持一致。
- **命名规范**：系统表 `sys_` 前缀，关联表 `sys_a_b`，索引 `idx_`、唯一键 `uk_`。
- **字符集 / 引擎**：统一 `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci`。
- **应用层**：本项目已具备统一响应、全局异常处理（`GlobalExceptionHandler` 统一处理 404 等）、MyBatis-Plus 逻辑删除与乐观锁等通用能力，新功能请复用，不要各写一套。

## 10. 附：如何自行复现"合并脚本"的生成与校验

如果将来又新增了若干迁移脚本，需要重新生成合并脚本，思路如下（与本次完全一致）：

1. 准备一套与生产同版本的 **MariaDB**。
2. 新建空库 `bml_system`，按**语义版本顺序**依次执行 `db/migration` 下全部 `V*.sql`。
3. 用 `mysqldump` 导出最终结构（`--no-data`）与数据（`--no-create-info --complete-insert`）。
4. 套上"建库 / 建账号 / 会话设置"头部，把 `CREATE TABLE` 改为 `CREATE TABLE IF NOT EXISTS`，时间戳统一为 `NOW()`，按业务分组加注释。
5. **回归校验**：把合并脚本在另一个全新空库执行，与"迁移真库"做逐表结构 + 逐行数据 `diff`，必须完全一致且 0 报错。
